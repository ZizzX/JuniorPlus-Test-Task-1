## ARCHITECTURE — FSD (Feature-Sliced Design)

Всегда соблюдай FSD:

* `app` — инициализация, провайдеры, глобальные конфиги
* `processes` — бизнес-процессы (если есть)
* `pages` — страницы
* `widgets` — крупные UI-блоки
* `features` — действия пользователя
* `entities` — бизнес-сущности
* `shared` — UI, lib, api, config

### Жёсткие правила:

* ❌ Никаких прямых импортов снизу вверх
* ❌ Никакой бизнес-логики в UI
* ❌ Никаких side-effects в entities
* ✅ Каждая фича делает **одно действие**

Если структура нарушается — предложи корректную.

## UI / LIBS

1. **Используй ТОЛЬКО уже установленные пакеты**
   Никаких самодельных велосипедов, если в проекте есть:

   * UI-kit
   * form-library
   * validation
   * state-manager

2. **Если требуется новая библиотека**:

   * объясни, **почему без неё хуже**
   * сравни минимум с 1 альтернативой
   * укажи влияние на bundle / DX / поддержку

## CODE STYLE & PRACTICES

* Чистые функции
* Явные типы
* Предсказуемые имена
* Минимум магии
* Ноль дублирования
* Ранний `return`

Каждая функция должна быть:

* легко тестируемой
* очевидной без комментариев
